# 个人中心 — 功能设计

## 目标

为博客社区新增个人中心页面，登录用户可查看自己的文章，按状态分组展示：草稿、待审核、已发布。

## 设计决策

- **后端极简**：只暴露一个 `GET /api/articles/my` 端点获取当前用户全部文章，不做状态筛选参数
- **前端分组**：拿到全部文章后在客户端按 `status` 字段用 `computed` 分组，切标签无需重新请求
- **分页**：后端保留分页支持，前端可一次取较大页（如 size=50）

## UI 布局

采用顶部标签页布局（布局 B）：

```
┌──────────────────────────────────┐
│  [头像] 用户名                    │
│         email · 注册时间          │
├──────────────────────────────────┤
│ [已发布(5)] [待审核(2)] [草稿(3)] │
├──────────────────────────────────┤
│ 文章卡片（标题/摘要/时间/统计）    │
│ 文章卡片                          │
│ ...                              │
│          < 分页 >                │
└──────────────────────────────────┘
```

每篇文章卡片展示：标题、摘要、创建时间、阅读量、点赞数、评论数、分类标签、操作按钮。

## 后端改动

### 1. handler/article.go — 新增 ListMyArticles

```
GET /api/articles/my
Headers: X-User-ID（网关注入）
Query: page, size
Response: { code, message, data: Article[], pagination: { total, page, page_size } }
```

复用现有 `service.ListMyArticles(authorID, page, size)` 和 `repository.ListByAuthor(authorID, page, size)`。

### 2. content-service/main.go — 注册路由

在认证路由组中添加：`router.GET("/api/articles/my", h.ListMyArticles)`

### 3. api-gateway/routes/router.go — 代理

在 `setupPrivateRoutes` 中添加：`router.GET("/api/articles/my", proxyTo("article"))`

## 前端改动

### 4. api/index.ts — 新增 API 方法

```typescript
getMyArticles(page = 1, size = 50) {
  return api.get('/articles/my', { params: { page, size } })
}
```

### 5. router/index.ts — 新增路由

```typescript
{ path: '/profile', name: 'profile', component: () => import('@/views/ProfileView.vue'), meta: { requiresAuth: true } }
```

### 6. components/Navbar.vue — 用户名改为链接

```html
<router-link to="/profile" class="nav-user">{{ userStore.userInfo?.username }}</router-link>
```

### 7. views/ProfileView.vue — 新建页面

核心结构：
- `onMounted` 调用 `articleApi.getMyArticles()`
- 三个 `computed`：`drafts` / `pendingArticles` / `publishedArticles` 按 status 过滤
- `currentTab` ref 控制当前激活标签
- `currentArticles` computed 根据 currentTab 返回对应数组
- 复用项目现有的 `ArticleCard.vue` 组件（若适用）或内联文章卡片

## 执行顺序

1. Backend: handler → content-service route → gateway proxy
2. Frontend: API → router → Navbar → ProfileView

## Phase 2: UX 优化

### 2.1 EditorView — 新增"保存草稿"按钮

**问题**: 新建文章时只有一个按钮，点击即提交审核，无单独的保存草稿入口。

**方案**: 将单一按钮拆分为两个独立按钮：

| 按钮 | 行为 |
|------|------|
| **提交审核**（蓝色主按钮） | 新建：创建文章 → 提交审核 → 跳转编辑页。编辑：更新文章 → 提交审核 → 更新状态 |
| **保存草稿**（白色次按钮） | 新建：创建文章 → 跳转编辑页（不提交审核）。编辑：更新文章 → 留在当前页 |

### 2.2 ProfileView — 草稿点击跳转编辑器

**问题**: 草稿标签页中点击文章主区域跳转到 `/article/:id`，应跳转到 `/editor/:id`。

**方案**: `goToArticle()` 按 `article.status` 分流——草稿跳 `/editor/:id`，其余跳 `/article/:id`。

### 2.3 NotificationView — 通知类型路由 + 字段修复

**问题 1**: `handleClick` 读取 `item.article_id`，但后端模型字段为 `source_id`（JSON 序列化名），导致通知点击永远无法跳转。

**问题 2**: 未按通知类型路由。`new_submission`（管理员审稿通知）和 `review_rejected`（驳回通知）均需要跳转到不同页面。

**方案**:

| 通知类型 | 跳转目标 |
|----------|----------|
| `new_submission` | `/admin/reviews`（管理员审稿页） |
| `review_rejected` | `/editor/:sourceId`（编辑文章页） |
| `comment` / `follow` / `like` | `/article/:sourceId`（文章详情页） |

### 2.4 Navbar — 未读通知红点标记

**问题**: 通知按钮显示红色数字胶囊，改为简洁的红色圆点。

**方案**: 去掉 `<span class="badge">` 内的数字文字，CSS 改为 `position: absolute; width: 8px; height: 8px; border-radius: 50%;`，定位在通知文字右上角。
